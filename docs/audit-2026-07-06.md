_Run 2026-07-06 · multi-agent workflow: 10 dimension finders → adversarial verification → synthesis · 67 agents, 3.3M tokens, 0 errors._
_Raw findings: 33 · Survived verification: 27 (all CONFIRMED) · Refuted: 6 · Distinct defects after dedup: ~20._

# Wildlight Imagery — Security & Reliability Audit (Final Report)

## Executive Summary

The storefront is fundamentally sound on its highest-stakes paths — Stripe/Printful webhook auth is HMAC/constant-time verified, money is integer cents, SQL is parameterized — but the audit surfaced **26 verified findings (all CONFIRMED)** that reduce to **20 distinct defects** after deduplication. Overall risk posture is **moderate**: no active auth bypass or double-charge, but two **High** defects can silently oversell signed limited editions and destroy live/print-master image files, and a cluster of **Medium** webhook/reliability gaps can leave a paid order uncreated or mis-refunded. Distinct count by severity: **2 High, 8 Medium, 9 Low, 1 Info**.

---

## High

**1. Limited-edition oversell: sold-count uses `COUNT(rows)` while the cap guard sums `quantity` (units)**
High — `app/api/checkout/route.ts:97`, `lib/editions.ts:30`, `app/api/admin/artworks/[id]/route.ts:38,259` *(dedup: merges the medium `payments-webhooks` + high `pricing-integrity` findings — same root defect)*
Edition of 5, buyer A buys `quantity=3` (1 row); guard reads `sold = COUNT(oi.id) = 1`, buyer B's `quantity=3` passes (`1+3 ≤ 5`) → 6 numbered prints sold of 5; storefront shows "remaining 4," never sold-out.
**Fix:** replace every edition `COUNT(oi.id)::int` with `COALESCE(SUM(oi.quantity),0)::int` so both sides of `ed.sold + requested > ed.edition_size` are in units.

**2. `finalize` UPDATE-mode rollback deletes the LIVE web image and print master of an existing artwork**
High — `app/api/admin/artworks/bulk-upload/finalize/route.ts:268`
Admin re-uploads a master for a published artwork; `uploadPublic(webKey)` overwrites the live object, the `withTransaction` UPDATE then hits a Neon idle-drop and throws, and the catch runs `deletePublic(webKey)`/`deletePrivate(printKey)` unconditionally — deleting the live catalog image and print master while the row still points at them (staged source already gone → unrecoverable). The row-delete is guarded by `createdRowId != null`; the file deletes are not.
**Fix:** gate the canonical-key deletes to create mode only (`if (createdRowId != null)`); in update mode write to temp keys and atomically swap only after DB commit.

## Medium

**3. Partial refund from the Stripe dashboard marks the whole order `refunded`**
Medium — `app/api/webhooks/stripe/route.ts:447`
A $20 goodwill partial refund on a $200 order fires `charge.refunded`; handler sets `status='refunded'` with no `amount_refunded >= total_cents` check, which frees the numbered-edition slot (sold-count filters `NOT IN ('canceled','refunded')`) so that exact numbered print can resell, and a later second partial refund is dropped by the unique index.
**Fix:** only transition to `refunded` when `amount_refunded >= total_cents`; record partials as a distinct event/status that does not free the edition slot.

**4. Webhook `event_id` is consumed before processing — a first-delivery failure is never retried, leaving paid orders uncreated**
Medium — `app/api/webhooks/stripe/route.ts:57`
`checkout.session.completed` claims the `webhook_events` row, then a Neon idle-drop makes the order-creation `withTransaction` throw; the catch returns HTTP 200 so Stripe never retries and the deduped `event_id` blocks reprocessing — customer charged, no order/email/Printful, and no `needs_review` row to find.
**Fix:** treat completion (not receipt) as the idempotency point — on failure DELETE the claimed row (or gate dedupe on `processed_at IS NOT NULL`) and return 5xx; and/or wrap the order txn in `withConnRetry`.

**5. Large-but-valid cart silently fails checkout — `cart_json` exceeds Stripe's 500-char metadata limit**
Medium — `app/api/checkout/route.ts:203` *(dedup: merges the two identical medium findings)*
Body schema allows 50 lines but the whole cart is serialized into one metadata value (`cart_json: JSON.stringify(lines)`); ~15–18 lines exceed 500 chars, `sessions.create` throws, customer gets a generic 502 `checkout_init_failed` well under the advertised 50-item ceiling.
**Fix:** carry `quantity` in the existing `checkout_intents` snapshot (`route.ts:251-262`) and have the webhook read from there; keep only a tiny intent reference in metadata.

**6. Login rate limiter is check-then-act (TOCTOU) — per-email/per-IP fail cap bypassable by concurrent bursts**
Medium — `lib/login-rate-limit.ts:31`
`checkLoginAttempts()` is a lock-free SELECT run before bcrypt; the failure INSERT happens after, so 50 concurrent admin-login POSTs all read `count=0`, all pass, and all run cost-12 bcrypt (~4,800 guesses/day vs the intended ~480), also defeating the pre-bcrypt CPU short-circuit.
**Fix:** make the gate atomic like `lib/rate-limit.ts:73-76` — INSERT first inside a `withTransaction` guarded by `pg_advisory_xact_lock`, then SELECT and block over cap.

**7. Anyone can silently re-subscribe an opted-out address (unauthenticated unsubscribe bypass)**
Medium — `app/api/subscribe/route.ts:37`
The "cheap re-subscribe" branch keys only on `confirmed_at` (SELECT omits `unsubscribed_at`) and runs before the rate limiter; `POST /api/subscribe {email}` for a previously-unsubscribed address unconditionally clears `unsubscribed_at`, silently re-adding the victim to every broadcast with no double-opt-in — a CAN-SPAM/GDPR opt-out defeat.
**Fix:** select `unsubscribed_at`; if previously opted out, route through the double-opt-in path (fresh confirm_token + email) and only clear `unsubscribed_at` inside `/confirm` on click.

**8. `ai-draft` Anthropic route has no rate limit — the one unbounded model-cost surface**
Medium — `app/api/admin/artworks/[id]/ai-draft/route.ts:41`
Every sibling AI route enforces a per-admin cap (generate 30/hr, synthesize 10/hr, etc.) explicitly as the stolen-cookie anti-cost control, but `ai-draft` has none; a loop of vision requests drains `ANTHROPIC_API_KEY` and, on the URL-fetch fallback, stacks a 25 MB R2 fetch + 100 Mpx sharp decode per call.
**Fix:** add `recordAndCheckRateLimit('artwork-ai-draft', session.email, 3600, ~120)` before `draftArtworkMetadata()`, sized to cover the 50-image bulk run.

**9. `withConnRetry` (built for WILDLIGHT-1) is wired into 1 route only — every server-rendered read path uses bare `pool.query`**
Medium — `app/admin/page.tsx:97`; `app/(shop)/shop/page.tsx:25`, `.../artwork/[slug]/page.tsx:47`, `.../collections/page.tsx:16`, `app/(shop)/journal/page.tsx:49`, `app/(shop)/portfolio/page.tsx:41`
The retry helper exists (`lib/db.ts:95`) but only `studio/draft/route.ts` uses it; a Neon idle-drop on the admin dashboard's 8-query `Promise.all` or any shop render throws an uncaught 500 — the direct cause of WILDLIGHT-1 (84) and WILDLIGHT-6 (2).
**Fix:** wrap the hot read-only SELECTs in `withConnRetry` (all single-statement, safe to retry), or fold retry into a shared read helper.

**10. Homepage swallows a transient DB drop into an empty wall that ISR caches for 60s (and logs it to console, never Sentry)**
Medium — `app/(shop)/page.tsx:60`
With `revalidate = 60`, a Neon idle-drop during a background revalidation makes the wall query throw; the catch renders the empty "wall is being hung" state via `console.error` (no `captureConsole` → invisible in Sentry) and Next caches that empty HTML for every visitor for up to 60s after the DB has recovered.
**Fix:** wrap the query in `withConnRetry`, switch to `logger.error`, and rethrow-on-empty so ISR keeps the last good cached wall.

**11. `requireAdmin` throws a raw `Response`, which App Router route handlers don't unwrap → 500 instead of 401 (root of WILDLIGHT-D)**
Medium — `lib/session.ts:93`; `app/api/admin/integrations/health/route.ts:6`
Next.js (unlike Remix) treats a thrown `Response` as an unhandled error, so all 42 catch-less admin routes return 500 (not 401) on an unauth/expired-session hit and forward it to Sentry as recurring noise (WILDLIGHT-D `ec.with`). Fails closed, but wrong status + drowns real errors.
**Fix:** throw a typed `AuthError` (or return a discriminated result) and translate per-handler, or provide a `withAdmin()` wrapper returning a 401 `NextResponse`.

## Low

**12. Concurrent checkouts can oversell an edition — cap is checked with no reservation**
Low — `app/api/checkout/route.ts:84`
Two buyers hit the last unit within the same second; both read `sold=49`, both pass, both pay → 51 of a 50-edition, and the webhook never re-checks the cap.
**Fix:** re-validate the cap inside the webhook order txn (`SELECT … FOR UPDATE`) and route an over-cap order to `needs_review`; keep the checkout check as a UX pre-filter.

**13. `getStripe()` memoizes the Stripe client across a test/live key change**
Low — `lib/stripe.ts:60`
When a `STRIPE_TEST_MODE_UNTIL` window expires inside a warm lambda, `getStripeConfig()` reports live while the cached `_client` still uses the TEST secret — the browser gets a live publishable key but the Checkout Session is created in test mode.
**Fix:** key the cached client by the resolved secret (rebuild when `pick().secret` changes), or drop memoization on this low-QPS path.

**14. Refund retry from `needs_review` re-invokes Stripe with no idempotency key**
Low — `app/api/admin/orders/[id]/refund/route.ts:26`
`needs_review` is not excluded from the atomic re-claim and `stripe.refunds.create` has no `idempotencyKey`; a retry after a transient commit failure re-calls Stripe on the already-refunded intent → "charge already refunded" → `needs_review` bounce (no double refund; self-heals via `charge.refunded`).
**Fix:** pass `idempotencyKey: refund_${id}` and exclude `needs_review` from the re-claim (or gate the Stripe call on an existing refund).

**15. `getClientIp` trusts client-supplied IP headers (`cf-connecting-ip` first) — per-IP rate-limit keys are attacker-controlled**
Low — `lib/rate-limit.ts:24`
On Vercel (no Cloudflare fronting), `cf-connecting-ip` is fully client-set; rotating it per request yields a fresh bucket every time, nullifying the per-IP cap on subscribe/contact and the login per-IP dimension (per-email cap still holds for admin login).
**Fix:** derive the IP from a platform-controlled source (`x-vercel-forwarded-for`, right-most untrusted hop); only trust `cf-connecting-ip` if actually behind Cloudflare.

**16. Money formatted inline with `/100).toFixed(2)` for Printful, bypassing `lib/money.ts`**
Low — `app/api/webhooks/stripe/route.ts:344`; `lib/printful-sync.ts:76`
No drift today (inputs are integer cents), but this is the one cents→string path outside the `money.ts` contract, so a future float/pre-summed amount will silently round with no single choke point. *(One verifier voted REFUTED as "style-only," but the convention violation and latent-drift risk are real.)*
**Fix:** add `centsToDecimalString(cents)` to `lib/money.ts` and use it in both spots.

**17. Presigned PUT URLs don't constrain object size — only the client-claimed `size` is validated**
Low — `lib/r2.ts:111`
Presigned `PutObjectCommand` signs only `Key`/`ContentType`, so a holder of a 900s URL can PUT a multi-GB body despite a declared 5MB `size`; `finalize`'s `getPrivateBuffer` then reads it into one Buffer → OOM the function, plus unbounded storage (public path lands oversized on the CDN).
**Fix:** sign an exact `ContentLength` bound to the validated `size`, and cap `getPrivateBuffer` with a streaming byte ceiling.

**18. `withTransaction` ROLLBACK runs unguarded in catch — on a dropped connection it masks the original error and releases a poisoned client**
Low — `lib/db.ts:46`
On a Neon idle-drop mid-transaction, `await client.query('ROLLBACK')` itself rejects, so callers/Sentry see "Connection terminated" from the ROLLBACK instead of the real error, `throw err` never runs, and `client.release()` is called with no error arg.
**Fix:** `try { await client.query('ROLLBACK') } catch(rbErr){ logger.warn(...) }`, always rethrow the original `err`, and `client.release(err instanceof Error ? err : undefined)`.

**19. `studio-reminder` cron: the final delivered-flag UPDATE is the one unguarded `pool.query` in an otherwise defensive handler**
Low — `app/api/cron/studio-reminder/route.ts:98`
Every other IO step is try/caught; the closing `UPDATE studio_reminders SET delivered` is not, so a Neon drop there 500s the handler after the email already sent, leaves `delivered=false`, and a 5xx cron retry can re-send the reminder. (Cron auth is correct/fail-closed.)
**Fix:** wrap the final UPDATE in try/catch + `withConnRetry`, mirroring the four preceding steps.

**20. `logger.warn/error` fire-and-forget the Sentry capture (no `await`, no flush) — events can be dropped on Vercel freeze**
Low — `lib/logger.ts:30`
`maybeSentry().then(s => s?.capture…)` is detached; Vercel can freeze the instance after the response resolves, so an error logged right before `return` (e.g. the studio-reminder cron catches, `session.version_lookup_failed`) may never reach Sentry. WILDLIGHT-2 is the benign flip side (every failed login → `captureMessage`).
**Fix:** `await` the capture + `Sentry.flush(2000)` before return in short-lived handlers; downgrade `auth.login_failed` to a breadcrumb so expected failures don't burn quota.

**21. Client-side Sentry never initializes, and its config lacks a `beforeSend` scrub**
Low — `sentry.client.config.ts:3`; `next.config.ts` (no `withSentryConfig`) *(dedup: merges the low `nextjs-framework` "never initializes" + info `secrets-config` "orphaned + no beforeSend")*
On `@sentry/nextjs ^10` / Next 16, the browser SDK loads only via a root `instrumentation-client.ts` (or `withSentryConfig` webpack injection) — neither exists, so `sentry.client.config.ts` is dead code and client-side checkout/Stripe-Elements errors are never reported. Separately, that config has no `beforeSend`, so if ever wired it would ship `/shop/orders/<token>` bearer URLs unredacted.
**Fix:** rename to root `instrumentation-client.ts` (keep the `Sentry.init` body) **and** give it the same shared `beforeSend` redactor as server/edge before enabling; optionally add `withSentryConfig` for source maps.

**22. Sentry `beforeSend` scrubs only the URL token, not `event.extra` or breadcrumbs where PII/secrets actually flow**
Low — `sentry.server.config.ts:12` (and byte-identical `sentry.edge.config.ts`)
`beforeSend` redacts only `event.request.url`/`query_string`; `lib/logger.ts:25` forwards `meta` straight into `extra`, so every login attempt stores `extra.email` + `extra.ip` in cleartext, and the Printful webhook token would survive in any breadcrumb — contrary to the comment's "breadcrumbs / transactions" claim.
**Fix:** a recursive `beforeSend`/`beforeBreadcrumb` (shared across all three configs) that redacts `token=`/`whsec_`/`sk_` and hashes PII, set `sendDefaultPii: false` explicitly, or stop forwarding raw `meta` into `extra`.

**23. No security response headers (CSP / X-Frame-Options / HSTS / X-Content-Type-Options)**
Low — `next.config.ts:6`
No `async headers()` anywhere, so admin/login/storefront ship none of these; there's admin-authored `journal-html` and a `dangerouslySetInnerHTML` mood script, so a CSP would be the backstop against any injected `<script>` (clickjacking already bounded by SameSite=Lax httpOnly cookie — this is defense-in-depth).
**Fix:** add `async headers()` returning `X-Content-Type-Options: nosniff`, `X-Frame-Options: DENY`, HSTS, and a Stripe/R2-scoped CSP (report-only first, tighten incrementally).

**24. Timing + email-backscatter oracle enumerates whether an address is a confirmed subscriber**
Low — `app/api/subscribe/route.ts:31`
A confirmed address short-circuits fast (no email); any other falls through to an **awaited** Resend round-trip, so response time classifies membership and probing a non-subscriber sprays them an unsolicited confirmation email; the per-`(ip,email)` cap doesn't stop enumeration across many addresses. The `/confirm` endpoint leaks the same via its distinguishable "Already subscribed" page.
**Fix:** don't `await` the Resend send (fire-and-forget/enqueue) so both branches return in comparable time; return an identical generic page from `/confirm` for non-existent/pending/confirmed ids.

## What Looks Solid

- **Webhook authentication is correct on both integrations** — Stripe HMAC-SHA256 body-signature verification and Printful's constant-time URL-token compare are in place; no finding challenged webhook *authenticity*, only idempotency/retry semantics (#4) and refund-status logic (#3).
- **Money is genuinely integer cents end-to-end** — every oversell/refund finding confirms values flow as authoritative Stripe/DB integer cents; the only formatting deviation (#16) has zero drift today and is latent-only.
- **The public rate-limiter pattern is well-built** — `lib/rate-limit.ts` correctly uses `pg_advisory_xact_lock` for atomic check-and-record; the login gap (#6) is precisely that login didn't adopt this existing, correct pattern.
- **Failures largely fail closed** — the checkout/metadata (#5), refund-retry (#14), and cron-auth paths deny or block rather than mischarge; the resilience work (`withConnRetry`, `isTransientPgError`, per-step try/catch in the studio cron) is the right shape and simply under-applied rather than absent.

## Cross-cutting coverage gaps (completeness critic)

These are seams the per-dimension split under-covers; several are effectively additional findings.

### G1. Global rate-limiting — public money & DB endpoints unprotected (app/api/checkout/route.ts, app/api/orders/by-session/[id]/route.ts)

**Why the per-dimension split missed it:** lib/rate-limit.ts is DB-backed and correctly wired into contact, subscribe, login, and the admin studio/voice routes — but NOT the two highest-value unauthenticated endpoints. POST /api/checkout is anonymous and, per request, runs 2 DB queries plus a live Stripe checkout.sessions.create; an attacker can loop it to burn Stripe API quota, spam checkout_intents rows, and drive Neon load. GET /api/orders/by-session/[id] runs a DB query and serves a 3s meta-refresh 'processing' page (self-amplifying loop) with no limiter. A per-dimension finder that inspects each route in isolation sees the limiter exists in the codebase and doesn't notice which routes omit it.

**Next step:** Add recordAndCheckRateLimit('checkout', getClientIp(req), ...) at the top of app/api/checkout/route.ts before the DB/Stripe calls, and a modest limiter on the by-session redirector. Do a route-by-route sweep enumerating every public POST/GET against the limiter to confirm intentional coverage rather than accidental omission.

### G2. Limited-edition oversell race (TOCTOU spanning checkout → Stripe webhook)

**Why the per-dimension split missed it:** app/api/checkout/route.ts lines 86-135 enforce the edition_size cap by counting sold order_items at Stripe-session-create time, with a comment claiming it is 'race-safe because we count at request time.' But the order row is not written there — it is written later by app/api/webhooks/stripe/route.ts, which contains NO edition_size re-check (grep confirms zero edition logic in the webhook). Two concurrent buyers of the last edition print both pass the session-create check (each sees sold=N-1), both pay, and both get order_items → the edition oversells. The 'money math' dimension likely marked the guard as present; the 'payments/webhook' dimension owns the writer but wasn't looking at editions. The cross-surface seam is exactly where the split loses the race.

**Next step:** Move (or duplicate) the edition-cap enforcement into the order-creation transaction in the Stripe webhook: SELECT the sold count FOR UPDATE / under an advisory lock keyed by artwork_id inside withTransaction, and route over-cap orders to needs_review + auto-refund rather than silently fulfilling. Correct the misleading 'race-safe' comment in checkout/route.ts.

### G3. Data-migration safety — schema.sql re-applied on every deploy, unversioned, no transaction (lib/migrate.ts, lib/schema.sql)

**Why the per-dimension split missed it:** npm run build = 'tsx lib/migrate.ts && next build', and migrate.ts does a single pool.query(sql) over the entire 27KB schema.sql with NO BEGIN/COMMIT and no migrations ledger. schema.sql contains data-mutating statements (UPDATE artworks at lines 192 and 513) and lock-taking DDL (ALTER ... SET NOT NULL = ACCESS EXCLUSIVE) that re-run on every deploy. A single failing statement mid-file leaves the DB half-migrated (earlier DDL auto-committed, later not) AND fails the deploy, since migrate gates next build. There is also no schema_migrations table to detect/skip already-applied steps. This 'runs on every build' blast radius is a cross-cutting reliability concern no per-file dimension owns.

**Next step:** Wrap the apply in an explicit transaction (BEGIN/COMMIT) so a failure rolls back cleanly, and introduce a lightweight applied-migrations ledger (or split into versioned, individually-guarded files) so idempotent-but-expensive DDL/UPDATEs don't re-execute on every deploy. Audit the two UPDATE artworks statements for true idempotency under concurrent deploys.

### G4. Dependency / supply-chain CVE sweep not run (package-lock.json)

**Why the per-dimension split missed it:** npm audit --omit=dev reports 18 vulnerabilities including 6 HIGH: undici (TLS certificate validation bypass via SOCKS5, HTTP header injection via Set-Cookie, response-queue poisoning via keep-alive socket reuse) plus a moderate uuid buffer-bounds bug. undici is the transitive fetch layer under the AWS S3 SDK (R2 presigning), Stripe, axios, and the Anthropic SDK — all of which make outbound TLS calls, so the TLS-bypass and header-injection classes are directly reachable. The 'Secrets, config & supply chain' dimension examined config files but there is no evidence a lockfile CVE scan was part of it.

**Next step:** Run npm audit fix (or targeted overrides pinning undici >=7.28 and uuid >=11.1.1), re-run typecheck/build, and add an npm audit / dependency-scan step to CI so transitive CVEs surface on future installs.

### G5. No HTTP security response headers anywhere (next.config.ts, middleware.ts)

**Why the per-dimension split missed it:** next.config.ts defines redirects() but no headers(), and middleware.ts only does host rewrites — so there is no Content-Security-Policy, X-Frame-Options/frame-ancestors, HSTS, or X-Content-Type-Options on any response. The admin panel (host-isolated but cookie-authenticated) has zero clickjacking protection, and the storefront has no CSP to blunt a stored-content or third-party-script XSS. The 'Next.js 16 framework correctness' dimension (only 5 files) focused on routing/RSC behavior, not the absence of a security-headers layer.

**Next step:** Add a headers() block in next.config.ts (or set headers in middleware) applying at minimum X-Frame-Options: DENY / frame-ancestors 'none' on /admin, HSTS, X-Content-Type-Options: nosniff, and a starter CSP; verify the Stripe Elements iframe and R2 image host stay allowlisted.

### G6. CSRF defense-in-depth (requireSameOrigin) applied to only 4 of 41 mutating handlers

**Why the per-dimension split missed it:** lib/origin-check.ts exists as an explicit defense-in-depth layer over SameSite=Lax, but grep shows it is imported by only 4 routes (admin/artworks, admin/artworks/[id], .../variants/[variantId], admin/wall). The other ~35 mutating POST/PATCH/DELETE handlers — including admin/orders/[id]/refund, resubmit, note, subscribers/broadcast, password, collections, all journal and voice-training routes — rely on the Lax cookie alone. The origin-check.ts header comment itself justifies the layer as protection for a plausible apex-cookie-scope regression, yet it's applied inconsistently. A route-at-a-time finder sees each route as 'has auth' and never notices the pattern is only half-deployed.

**Next step:** Sweep every state-changing admin handler for a consistent requireSameOrigin() call (or centralize it in a shared admin-route wrapper / middleware) so the defense-in-depth layer is uniform rather than present on 10% of mutating routes.

### G7. Order state-machine concurrency across the four handlers that mutate order status

**Why the per-dimension split missed it:** Four independent handlers write order status — app/api/webhooks/stripe/route.ts (paid/refunded), app/api/webhooks/printful/route.ts (shipped/delivered/canceled/needs_review), app/api/admin/orders/[id]/refund/route.ts, and app/api/admin/orders/[id]/resubmit/route.ts (bumps printful_attempt). The dimension split assigned webhooks and admin-order routes to different finders, so no one reviewed the concurrent interleavings: e.g. an admin refund racing a Stripe charge.refunded webhook (double-refund / status flip-flop), or a resubmit incrementing printful_attempt while an in-flight package_shipped for the prior attempt lands, or the by-session redirector vs the webhook both racing to materialize the order. The printful_attempt guard handles the stale-shipment case but the refund/resubmit/refund-webhook triad is unreviewed as a whole.

**Next step:** Do a dedicated cross-handler race review of the order status machine: enumerate every (status_from → status_to) writer, confirm each transition is guarded by a conditional UPDATE (WHERE status = expected) inside withTransaction, and add a test/matrix for concurrent refund+resubmit and refund-route + refund-webhook on the same order.

### G8. Prompt-injection as an attack class: untrusted web-search / EXIF content flowing into LLM calls (lib/studio.ts, lib/ai-draft.ts, lib/anthropic-image.ts, lib/voice-trainer.ts)

**Why the per-dimension split missed it:** Stored-XSS is well-mitigated — journal/broadcast HTML is DOMPurify-sanitized at write time (lib/journal-html.ts) and the render page relies on that. But the cron studio-reminder and the studio draft pipeline feed untrusted external content (researchSeoTrends web_search results, scraped titles, uploaded-image EXIF) into Anthropic prompts, and the model output then drives auto-drafted marketing copy and image generation. The 'AI studio' dimension likely reviewed functionality, not the adversarial chain where a poisoned search result or crafted EXIF field steers the model (content-integrity, tool/cost abuse, or prompt-leak of the voice profile). This attack class sits across studio + email + cron surfaces.

### G9. Accessibility of the money flow (app/(shop)/shop/cart, checkout, app/api/orders/by-session/[id]/route.ts)

**Next step:** Review the cart → checkout → confirmation path for keyboard traversal and screen-reader error announcement (aria-live on the checkout error region), and replace the meta-refresh processing page with a polite polling/aria-live update or an accessible client-side poll.
